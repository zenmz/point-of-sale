## 1. Latar Belakang & Masalah

UMKM dan bisnis multi-cabang di Indonesia butuh sistem kasir yang:
- Bekerja lintas tipe bisnis (retail, kafe, jasa).
- Tetap melayani transaksi saat internet tidak stabil/putus.
- Mengelola banyak cabang dengan laporan tergabung.
- Mendukung metode bayar lokal (QRIS, e-wallet, transfer).

Solusi yang ada sering: mahal, online-only (mati saat internet putus), atau
hanya untuk satu tipe bisnis.

## 2. Tujuan Produk

**MZ POS** — aplikasi POS berbasis web (PWA) yang general, multi-toko, dan
tahan offline untuk transaksi.

### Tujuan terukur (success metrics)
- Kasir dapat menyelesaikan 1 transaksi tunai **< 15 detik**.
- Penjualan tetap berjalan **100%** saat offline; sync otomatis saat online.
- 0 transaksi dobel setelah sync (idempotent).
- Laporan penjualan harian tersedia **real-time** saat online.

### Non-tujuan (Out of Scope)
- Integrasi akuntansi eksternal / payroll.
- E-commerce / marketplace.
- Sync engine CRDT full offline-first.
- Server lokal per toko.

## 3. Persona Pengguna

| Persona | Peran | Kebutuhan utama |
|---------|-------|-----------------|
| **Kasir** | Layani transaksi | Checkout cepat, terima bayar, cetak struk, buka/tutup shift |
| **Admin Toko** | Kelola 1 cabang | Kelola produk, stok, lihat laporan toko |
| **Owner** | Kelola multi-cabang | Laporan gabungan, transfer stok, kelola user antar cabang |

## 4. User Stories (per fase)

### Fase 1 — MVP

- Sebagai **kasir**, saya login agar transaksi tercatat atas nama saya.
- Sebagai **kasir**, saya tambah produk ke keranjang lewat barcode/cari nama agar checkout cepat.
- Sebagai **kasir**, saya beri diskon per item / per nota.
- Sebagai **kasir**, saya terima bayar tunai dan sistem hitung kembalian.
- Sebagai **kasir**, saya terima bayar QRIS dan tandai lunas.
- Sebagai **kasir**, saya cetak / kirim struk ke pelanggan.
- Sebagai **kasir**, saya buka shift dengan kas awal dan tutup shift dengan rekap.
- Sebagai **admin**, saya CRUD produk, kategori, varian, harga, barcode.
- Sebagai **admin**, saya atur stok masuk/keluar dan stok berkurang otomatis saat jual.
- Sebagai **admin**, saya lihat laporan penjualan harian & produk terlaris.

### Fase 2 — Offline + Multi-toko

- Sebagai **kasir**, saya tetap jualan saat internet putus; transaksi tersimpan dan sync otomatis.
- Sebagai **kasir**, saya lihat indikator status sync (pending/sukses/gagal).
- Sebagai **owner**, saya kelola banyak cabang & user per cabang.
- Sebagai **owner**, saya lihat laporan gabungan lintas cabang dengan filter per toko.
- Sebagai **admin**, saya transfer stok antar cabang.
- Sebagai **admin**, saya lakukan stock opname/audit.

### Fase 3 — Lanjutan

- Sebagai **kasir**, saya catat member & poin loyalti saat transaksi.
- Sebagai **admin**, saya buat Purchase Order ke supplier & terima barang.
- Sebagai **admin**, saya buat promo (bundling, diskon kuantitas, happy hour).
- Sebagai **owner**, saya lihat dashboard analitik (tren, margin, prediksi stok).
- Sebagai **kasir**, saya pakai printer thermal, cash drawer, scanner.

## 6. Non-Functional Requirements

| ID | Requirement |
|----|-------------|
| NFR-01 | PWA installable, jalan di browser modern (Chrome/Edge/Safari) |
| NFR-02 | Transaksi offline tidak hilang saat tab/app ditutup |
| NFR-03 | Respon UI checkout < 200ms per aksi |
| NFR-04 | Data sensitif (password) di-hash (bcrypt/argon2) |
| NFR-05 | Komunikasi API via HTTPS |
| NFR-06 | Sync idempotent — tidak ada transaksi dobel |
| NFR-07 | Audit trail untuk pergerakan stok |

## 7. Tech Stack

- **Frontend:** React + Vite, PWA (service worker), IndexedDB (Dexie)
- **Backend:** Go (chi/gin), PostgreSQL
- **Auth:** JWT, RBAC

## 8. Asumsi & Risiko

| Risiko | Mitigasi |
|--------|----------|
| Konflik stok saat banyak kasir offline jual barang sama | Server jadi sumber kebenaran; tandai konflik untuk admin |
| Integrasi QRIS dinamis butuh provider PJSP | Fase 1 mulai dari QR statis; QR dinamis via provider di fase lanjut |
| Sync gagal/partial | Antrian retry + status per transaksi |
